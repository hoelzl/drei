# Sixth editor slice: undo

**Status:** ready — architecture gate: undo history is **session state** (like the kill ring — design 0002's "identity holds mutable session state" precedent: `_kill_ring`, `_yank_*`). No new ports; `BufferValue` itself unchanged.

**Goal:** `C-/` (`undo`, also `C-x u` and `C-_`) reverses text-changing commands, most-recent first; a second consecutive undo descends further; any event-emitting non-undo command clears the redo direction (Emacs's undo-in-region/redo semantics simplified — see deviations).

## Behavior contract (Drei-specified, Emacs-informed)

- Every **text-changing command** (`InsertText` with text, `KillLine`/`KillRegion` with events, `Yank`, `YankPop` with events) pushes an **undo group** on the session's undo stack: enough to restore `(text, point)` — the inverse patch `(before_text_span, after_text_span, point_before, point_after, mark_before, mark_after)`.
- `C-/` pops the newest group and **applies the inverse**: restores the prior text span and point; pushes the forward direction onto the same stack (Emacs pushes redo entries onto `buffer-undo-list` — Drei: a single stack, direction tracked by the entries themselves, exactly like Emacs's `undo-more` walking).
- **Consecutive undos descend** the history one group at a time. The descent is broken only by a **text-changing command** (which pushes a fresh group and truncates the redo tail); point motion and other text-preserving commands between undos do NOT break it (probed rule — see "Descent rule" below).
- **Redo** (Emacs: `undo` after direction flip, or `C-?`/`undo-redo` in 28+): Drei implements redo as **undo of the undo** — after a descent, the first non-undo command clears the redo-able tail, matching Emacs's "new edit breaks the redo chain" (`L1` probe: undo pushes inverse entries; a new edit makes them unreachable).
- **Nothing to undo** (empty stack): silent no-op, no event (deviation — Emacs signals "No further undo information", verified in batch probe; same no-echo-error rationale as empty kill).
- Undo restores the **mark** from the group (Emacs records marker adjustments in the undo list; a killed region's mark is restored with the text — probe shows `(t . 0)` flag entries; marker restoration is the observable piece).
- **Modified flag:** undo sets/clears modified by comparing to the saved state — Emacs records `(t . 0)` entries (probe: `U2 MOD=nil` after undoing back to a saved/clean state). Drei: each group stores `modified_before`; undo restores it. (Emacs's exact "undo to saved point clears modified" is pinned by the differential below.)
- **Kill ring is NOT undone** (Emacs: the ring is global, undo doesn't pop it — verified: `L1` probe shows redo entries are separate from `kill-ring`).
- **Yank-pop after undo**: undo is event-emitting → clears yank-active (consistent with the existing rule).
- **Undo limit**: Emacs `undo-limit`/`undo-strong-limit` (default 80k/120k bytes). Drei: fixed capacity 100 groups, oldest dropped silently (deviation: configurable limits deferred; 100 ≫ any test horizon).

## Command / event surface

- `Undo` → `TextUndone(inverse: the group applied, groups_remaining: int)`; no-op → no event.
- `Redo` → `TextRedone(...)`? **Decision: NO separate Redo command.** Emacs exposes redo via `undo` itself (direction flip on non-consecutive undo) or `undo-redo` (28+). Drei scope: `C-/` only, with Emacs's direction-flip semantics: `undo` after `undo` descends; `undo` after an edit undoes the edit. True redo via a second undo after an intervening undo-descent is Emacs-arcane (the "undo the undo" trick works in Drei naturally: the descent groups are themselves undone... no — Drei pops, it doesn't re-push. **Simplest honest rule:** the undo stack holds applied-group inverses; undoing re-pushes the forward patch as a redoable entry; a fresh edit truncates redo entries. This IS Emacs's model minus the interactive message noise.)
- Keys: `C-/` (`\x1f`), `C-x u` prefix, `C-_` (`\x1f` — same byte as C-/ in terminals, one decode entry).
- Events: `TextUndone(old_text, new_text, point_before, point_after, mark: int | None)` and `TextRedone(...)` as separate types keeps the fold unambiguous (mirrors the RegionKilled split lesson from slice 5's review). **The mark field is required**: undo can resurrect a mark with no `MarkSet` event, so the slice-5 mark-fold property's event set extends with `TextUndone`/`TextRedone` (mark = their `mark` field); without this the mark would be underivable from the transcript after an undo.
- **Existing-property amendment (explicit):** the modified-flag property's rule extends — `TextUndone`/`TextRedone` set the expectation from the group's stored `modified_before` (undo) / `modified_after` (redo), NOT unconditionally True; the other arms are unchanged.

## Emacs evidence (probed vs pinned 29.3, batch)

- `(insert)` records `(start . end)` + `(t . 0)` modified-flag snapshot in `buffer-undo-list`; explicit `(undo-boundary)` inserts `nil`.
- Batch `undo` AMALGAMATES groups across explicit boundaries (`U1` undid two boundary-separated inserts at once) — grouping is command-loop driven, **batch-unverifiable**; pinned by unit/property tests (one group per Drei command = the interactive equivalent). **This also makes descent semantics batch-unverifiable** (U1 consumed the whole history, so motion-between-undos can't be probed in batch — the descent rule must be Drei-specified, see below).
- Fresh-edit truncation IS batch-verifiable: after `undo` + fresh `insert` + `undo`, the text returns to the pre-insert state and the previously undone groups are NOT resurrected (`U3="ab"`, `U4="ab"` — no resurrection).
- Undo restores the modified flag when returning to the clean state (`U2 MOD=nil` probe).
- Exhausted history signals "No further undo information" (error) — Drei: silent no-op (deviation).

**Descent rule (Drei-specified, batch-unverifiable in Emacs):** the descent breaks on **text-changing** commands only — point motion between undos does NOT break the descent (matches interactive Emacs, where motion doesn't reset the undo walk; batch can't confirm, recorded). `SetMark`/`CopyRegionAsKill`/`ExchangePointAndMark`/`KeyboardQuit` are event-emitting but text-preserving — they also do not break the descent (same rule class). Any event clears yank-active as before.

## Implementation order

1. Session undo stack: `_undo_groups: list[_UndoGroup]` (frozen record: inverse patch + point/mark/modified before & after), capacity 100; push on every text-changing dispatch arm (insert/kill-line/kill-region/yank/yank-pop); truncate redo tail on push.
2. `Undo` command + `TextUndone`/`TextRedone` events (TDD): apply inverse, restore point/mark/modified, direction flip semantics.
3. Keys: `\x1f` → `C-/` in decode/_CONTROL_KEYS; `C-x u` in `_PREFIX_COMMANDS`.
4. Property tests: strategy gains `Undo`; invariants — (a) N text-changes then N undos restores the initial text/point/modified; (b) undo-bounds: stack length never exceeds 100; (c) replay compares text/point/mark/modified AND remaining stack depth; (d) fresh edit after descent truncates redo (a redo after a fresh edit never resurrects truncated state); (e) modified-flag arms: `TextUndone`/`TextRedone` set modified from the group's stored value.
5. TermVerify scenario: type "ab", `C-/` removes "b"... per-command grouping: "ab" is two InsertText commands → one undo removes "b", second removes "a" (pinned in-process too). Frame evidence: insert via keys, `C-/` twice, frame shows the reverted text.
6. Emacs differential: batch eval — insert, undo → empty text + MOD=nil (probed `U2`); insert after undo → redo tail truncated (L1-style: new group unreachable). Parity required on text/point/modified after single undo; grouping/count deviations recorded (batch-unverifiable).
7. Docs: README, registry rows (grouping batch-unverifiable, no-further-undo no-op, capacity 100, no undo of the kill ring), plan status.

## Acceptance criteria

- Full quality gate green; coverage ratchet 100%.
- Undo round-trip pinned by unit + property + differential (single-undo text/point/modified parity vs 29.3).
- TermVerify: `C-/` reverts keyed insertions frame-by-frame through ConPTY (`\x1f` is an ordinary control byte — no delivery risk like NUL).
- Registry rows for the three deviations; no silent drift.

## Risks and decisions

- Grouping is the deepest Emacs-behavior gap (command-loop driven, batch-unverifiable) — mitigated by documenting one-group-per-command as the deliberate interactive-equivalent rule, property-tested.
- Redo semantics chosen as "undo-the-undo with truncation on fresh edit" — the Emacs model minus message noise; if a reviewer wants `undo-redo` (`C-?`) as a separate command, that's a follow-up slice, not this one.
