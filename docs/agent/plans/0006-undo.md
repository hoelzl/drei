# Sixth editor slice: undo

**Status:** ready — architecture gate: undo history is **session state** (like the kill ring — design 0002's "identity holds mutable session state" precedent: `_kill_ring`, `_yank_*`). No new ports; `BufferValue` itself unchanged.

**Goal:** `C-/` (`undo`, also `C-x u` and `C-_`) reverses text-changing commands, most-recent first; a second consecutive undo descends further; any event-emitting non-undo command clears the redo direction (Emacs's undo-in-region/redo semantics simplified — see deviations).

## Behavior contract (Drei-specified, Emacs-informed)

- Every **text-changing command** (`InsertText` with text, `KillLine`/`KillRegion` with events, `Yank`, `YankPop` with events) pushes an **undo group** on the session's undo stack: enough to restore the buffer state — the inverse patch `(start, removed_text, inserted_text)` plus `(point_before, point_after, mark_before, mark_after, modified_before, modified_after)` (before = arm entry, before the frozen-`BufferValue` adjustment runs).
- `C-/` pops the newest group and **applies the inverse**: restores the prior text span and point; pushes the forward direction onto the same stack (Emacs pushes redo entries onto `buffer-undo-list` — Drei: a single stack, direction tracked by the entries themselves, exactly like Emacs's `undo-more` walking).
- **Consecutive undos descend** the history one group at a time. **Any event-emitting non-undo command breaks the descent (motion included — matching Emacs's `last-command` gating, where any command flips direction); text-changing commands additionally push a fresh group.** Drei gates on event-emitting, Emacs on any command — silent no-ops (empty-ring yank, no-mark kill-region) don't flip direction in Drei; recorded as a batch-unverifiable deviation.
- **Redo:** Drei implements redo as **undo of the undo** — descending re-pushes the forward patch as a redoable entry. **A fresh edit truncates the redo tail — this is a Drei deviation, owned as one:** stock Emacs keeps redo entries reachable (a continued undo-more first undoes the new edit, then redoes the buried undo, then keeps descending); Drei's truncation is the deliberate simplification (registry row).
- **Nothing to undo** (empty stack): silent no-op, no event (deviation — Emacs signals "No further undo information", verified in batch probe; same no-echo-error rationale as empty kill).
- Undo restores the **mark** from the group (Emacs records marker adjustments in the undo list; a killed region's mark is restored with the text — probe shows `(t . 0)` flag entries; marker restoration is the observable piece).
- **Modified flag:** undo sets/clears modified by comparing to the saved state — Emacs records `(t . 0)` entries (probe: `U2 MOD=nil` after undoing back to a saved/clean state). Drei: each group stores `modified_before`; undo restores it. (Emacs's exact "undo to saved point clears modified" is pinned by the differential below.)
- **Yank-pop after undo**: undo is event-emitting → clears yank-active (consistent with the existing rule), so a post-undo `M-y` no-ops — Emacs-consistent, batch-unverifiable, pinned in-process. The yank-pop group's inverse (delete `[start, start+len(new))`, insert old at `start`) composes with the existing delete+insert mark adjustment.
- **Kill ring is NOT undone** (Emacs: the ring is global, undo doesn't pop it). Focused pin: kill, undo, yank → yanked text == killed text.
- **Undo limit**: Emacs `undo-limit`/`undo-strong-limit` (default 80k/120k bytes). Drei: fixed capacity 100 groups, oldest dropped silently (deviation: configurable limits deferred; 100 ≫ any test horizon).

## Command / event surface

- `Undo` → `TextUndone(inverse: the group applied, groups_remaining: int)`; no-op → no event.
- `Redo` → `TextRedone(...)`? **Decision: NO separate Redo command.** Emacs exposes redo via `undo` itself (direction flip on non-consecutive undo) or `undo-redo` (28+). Drei scope: `C-/` only, with Emacs's direction-flip semantics: `undo` after `undo` descends; `undo` after an edit undoes the edit. True redo via a second undo after an intervening undo-descent is Emacs-arcane (the "undo the undo" trick works in Drei naturally: the descent groups are themselves undone... no — Drei pops, it doesn't re-push. **Simplest honest rule:** the undo stack holds applied-group inverses; undoing re-pushes the forward patch as a redoable entry; a fresh edit truncates redo entries. This IS Emacs's model minus the interactive message noise.)
- Keys: `C-/` (`\x1f`), `C-x u` prefix, `C-_` (`\x1f` — same byte as C-/ in terminals, one decode entry).
- Events: `TextUndone(start: int, removed_text: str, inserted_text: str, point_before: int, point_after: int, mark_before: int | None, mark_after: int | None)` and `TextRedone(same fields)` as separate types keeps the fold unambiguous (mirrors the RegionKilled split lesson from slice 5's review). **Both mark fields and the span position are required**: undo of a backward kill-region re-inserts at `lo` while restoring `point_before`, and undo of a kill-region resurrects a cleared mark with no `MarkSet` event — so the slice-5 mark-fold property's rule gains arms: `TextUndone`/`TextRedone` set mark-setness from their `mark_after` field (and the text fold replays the span patch at `start`). Without this the mark would be underivable from the transcript after an undo.
- **Existing-property amendment (explicit):** the modified-flag property's rule extends — `TextUndone`/`TextRedone` set the expectation from the group's stored `modified_before` (undo) / `modified_after` (redo), NOT unconditionally True; the other arms are unchanged. The parity doc's rule paragraph ("any text-changing event sets modified") gets the matching sentence: undo/redo restore the flag from the group rather than setting it.

## Emacs evidence (probed vs pinned 29.3, batch)

- `(insert)` records `(start . end)` + `(t . 0)` modified-flag snapshot in `buffer-undo-list`; explicit `(undo-boundary)` inserts `nil`.
- Batch `undo` AMALGAMATES groups across explicit boundaries (`U1` undid two boundary-separated inserts at once) — grouping is command-loop driven, **batch-unverifiable**; pinned by unit/property tests (one group per Drei command = the interactive equivalent). **This also makes multi-undo descent semantics batch-unverifiable** (U1 consumed the whole history — the direction-flip rule is Drei-specified from the interactive model, recorded).
- Stock-Emacs redo reachability (undo-more redoes buried undos after a fresh edit) is documented from the manual, not probed — batch amalgamation obscures it; Drei's truncation is an owned deviation.
- Undo restores the modified flag when returning to the clean state (`U2 MOD=nil` probe).
- Exhausted history signals "No further undo information" (error) — Drei: silent no-op (deviation).

## Implementation order

1. Session undo stack: `_undo_groups: list[_UndoGroup]` (frozen record: inverse patch + point/mark/modified before & after), capacity 100; push on every text-changing dispatch arm (insert/kill-line/kill-region/yank/yank-pop); truncate redo tail on push.
2. `Undo` command + `TextUndone`/`TextRedone` events (TDD): apply inverse, restore point/mark/modified, direction flip semantics.
3. Keys: `\x1f` → `C-/` in decode/_CONTROL_KEYS; `C-x u` in `_PREFIX_COMMANDS`.
4. Property tests: strategy gains `Undo`; invariants — (a) N text-changes then N undos restores the initial text/point/modified; (b) undo-bounds: stack length never exceeds 100 (oldest dropped mid-descent shortens the deepest history; exhaustion degrades to the silent no-op); (c) replay compares text/point/mark/modified AND remaining stack depth; (d) fresh edit after descent truncates redo (a redo after a fresh edit never resurrects truncated state); (e) modified-flag arms: `TextUndone`/`TextRedone` set the expectation from the group's stored flags; (f) mark-fold coherence STILL holds with `Undo` in the strategy (the extended fold arms from the event bullet); (g) kill-ring untouched by undo (ring derivation unchanged).
5. TermVerify scenario: type "ab", `C-x u` removes "b" (`C-x u` drives the scenario — `\x1f` byte delivery on the Windows console is unverified; probe it live first, and if it fails, `C-x u` stays the scenario and `\x1f` is proven in-process — the slice-5 NUL lesson). Frame evidence: insert via keys, undo twice, frame shows the reverted text.
6. Emacs differential: batch eval — insert, undo → empty text + MOD=nil (probed `U2`); insert after undo (truncation arm: fresh insert then undo returns to pre-insert state, previously undone groups not resurrected — probed `U3`/`U4`). Parity required on single-undo text/point/modified; grouping/descent/truncation differences recorded as deviations (batch-unverifiable or owned simplifications).
7. Docs: README, registry rows (grouping batch-unverifiable, no-further-undo no-op, capacity 100, no undo of the kill ring, **fresh-edit truncation** — Emacs keeps redo reachable via undo-more, **event-emitting vs any-command descent gating** — silent no-ops don't flip direction in Drei, parity-doc modified-flag paragraph sentence), plan status.

## Acceptance criteria

- Full quality gate green; coverage ratchet 100%.
- Undo round-trip pinned by unit + property + differential (single-undo text/point/modified parity vs 29.3).
- TermVerify: `C-/` reverts keyed insertions frame-by-frame through ConPTY (`\x1f` is an ordinary control byte — no delivery risk like NUL).
- Registry rows for the three deviations; no silent drift.

## Risks and decisions

- Grouping is the deepest Emacs-behavior gap (command-loop driven, batch-unverifiable) — mitigated by documenting one-group-per-command as the deliberate interactive-equivalent rule, property-tested.
- Truncation is an owned deviation (Emacs keeps redo reachable via undo-more after a fresh edit); the simpler model is deliberate. If a reviewer wants `undo-redo` (`C-?`) or stock redo reachability, that's a follow-up slice, not this one.
- `\x1f` (C-//C-_) byte delivery on the Windows console is unverified — probe first (slice-5 NUL lesson); `C-x u` is the TermVerify fallback.
